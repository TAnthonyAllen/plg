## 1. What plg is, and the self-hosting chain

plg is the parser-generator. It reads a grammar (`*.g`) and emits TAWK source (`*.twk`),
which `tok` compiles to C++ (`*.C`/`*.h`).

**plg is self-hosting** — it generates its own parser:

```
plg.g  ──(plg)──▶  PLG.twk  ──(tok)──▶  PLG.C/.h  ──▶  plg binary
```

The Tawk transpiler is generated the same way (this is the recurring task):

```
Tawk.g ──(plg)──▶  Tawk.twk ──(tok)──▶  Tawk.C/.h  ──▶  tok
```

**The POP for the grammar sources is: `plg plg.g` regenerates today's plg.** If that
round-trips cleanly, the grammar files and the runtime are in sync. They are NOT fully in
sync yet — see §11.

⚠️ **Never run `plg plg.g` casually.** If the grammar sources are stale relative to a
hand-revised `PLG.twk`, it will overwrite `PLG.twk` with old output and lose work. Only run
it once you've confirmed the grammar describes the current runtime.

## 2. File locations (and don't-move rules)

- **plg's own grammar sources live in `Parse/Grammar/`** — git-tracked, this is their HOME,
  **do not move them**: `plg.g`, `plg.rtn`, `plg.act`, `action.g`, `parts.g`, `plgRules.g`,
  `PLGincludes`. (`Parse/BeforeRefactor/` holds an older untracked backup copy — ignore it.)
- **Tawk's grammar sources live in `Tokf/`**: `Tawk.g`, `Tawk.rtn`, `Tawk.act`, plus the
  included sub-grammars `parts.g`, `declare.g`, `expression.g`, `generate.g`, `keywords.g`
  and their `.act` action files. The Tawk test harness is `Tokf/Tests/` (its `Tawk.*` are
  symlinks back to `Tokf/`); `plg Tawk.g` is run there and writes `Tokf/Tests/Tawk.twk`.
- **plg externals: `Include/PLGrevision`** (this is the file formerly called `plg.ext`).
  External `PLGitem` / `PLGparse` / `PLGrule` / `PLGset` declarations live here. Anything
  that needs to be visible to *other* files at tok-compile time (overloads, aliases, method
  signatures) goes in the relevant `external X { }` block here.
- **`Include/tok.ext`** holds the `external Tawk extends PLGparse { }` block — Tawk's field
  declarations (state fields, PLGset/KeyTable fields) and method signatures. Pulled into
  `Tawk.twk` via `include includes` (→ `Tokf/includes` → PLGrevision + tok.ext).

⚠️ **Case-insensitive filesystem:** `plg.twk` and `PLG.twk` are the *same path*. Don't try
to keep both. (`Grammar/plg.twk` is a small legacy artifact, not the generated parser.)

## 3. Generated vs hand-maintained (critical distinction)

- **Generated — do not hand-edit (regen overwrites):** `PLG.twk` (from `plg.g` + insert
  `plg.rtn` + include `plg.act`), and `Tawk.twk` (from `Tawk.g` + its `.rtn`/`.act`). Any
  hand-edit here is a temporary stopgap that the next regen erases.
- **Hand-maintained runtime classes (the real source):** `PLGparse.twk`, `PLGrule.twk`,
  `PLGitem.twk`, `Alternative.twk`, `Element.twk`, `GroupBody`/`GroupStak`/etc. Edit these
  directly.
- **`.C`/`.h`** are tok output from the `.twk`. Regenerate with `tok X.twk plgDirectives`.

## 4. The action (`.act`) format — redesigned 2026-05-31

Each action entry:

```
ruleName[:tag] [defer]
<body lines …>
enD
```

- **`ruleName`** alone → option 1; **`ruleName:N`** → numbered option; **`ruleName:name`**
  → named option. The key passed to `getRule()` is `ruleName` + tag (e.g. `Block`,
  `Block2`, `Blockcast`).
- **`defer`** keyword after the tag → the body becomes the rule's `deferAction`
  (emitted as `void <Name><Parser>Act(PLGitem iTEM)`). Absent → `immediateAction`
  (emitted as `int <Name><Parser>Now(PLGitem iTEM)`).
- **`enD`** on its own line terminates the body. **It is `enD`, not `end`, deliberately** —
  `end` is a common body variable (e.g. `PoundCommand`'s `PLGitem end = state;`), and a
  plain `end` terminator collided with it. `enD` is case-distinct and never appears in body
  code. (`action.g`'s `ActionEnd` rule must use `'enD'`, not `'end'`.)
- One option per entry. The **old** format used `Name!`, a leading `.` for immediate, `|`
  (and combined forms `|.`, `||.`, `|||.`) to separate options, and `;` to terminate. The
  migration split every option into its own `:N` entry, reading the per-option
  immediate/defer status from whether that option had a leading `.`.

## 5. The attachActions / writeActions bootstrap

Two hand-written pieces in the runtime classes drive action codegen:

- **`PLGparse::attachActions(content)`** — a hand-parser for the `.act` format. For each
  entry it computes the rule key (`ruleName`+tag), reads the body to the `enD` line, and
  attaches it to `getRule(key).immediateAction` or `.deferAction` (wrapped in a
  `PLGitem`). Invoked from `IncludeplgNow`'s `.act` branch. **This is the bootstrap**: it
  lets the *current* plg ingest the new `.act` format so it can regenerate itself, before
  `action.g`'s grammar-driven path is enabled (§6).
- **`PLGrule::writeActions(parserName, output)`** + **`writeCaptures(output)`** — emit each
  action body as a method *inside* the generated class. `writeCaptures` injects one
  `PLGitem <label> = iTEM.children["<label>"];` per labelled element. **No parser-pointer
  injection** — actions are methods on the parser, so they reach state directly as members
  (this is why `iTEM.testParser.X` in old bodies becomes bare `X`).
- **`PLGrule::generate(parserName, output)`** — emits the `setRules()` wiring
  (`currentRule.immediate = <Name><parserName>Now;` etc.). Note `parserName` is **passed
  in** (the grammar base name, e.g. `"Tawk"`), not read from a `parentParser` field —
  `parentParser` is never assigned, and the owning parser's own `parserName` is the
  meta-parser's (`"PLG"`).

## 6. The action.g enablement story

`action.g` (rules `ActionEnd` / `ActionBody` / `ActionRule`, plus `OptionTag`) is the
*durable* action-parsing path — `plg.g` would `include(action.g)` so the generated parser
recognises action entries natively, firing `ActionRuleplgNow` to attach bodies to rules.

It was **commented out** of `plg.g` (`/* include(action.g) deferred … current parser
cannot ingest ActionEnd/ActionBody/ActionOption bodies */`) and `ActionRuleplgNow` was
disabled (`//currentRule.immediate = ActionRuleplgNow;` in `PLG.twk`). The 2026-05-31
redesign rewrites `action.g` for the new format. **Still pending** (named-options pass):
enable `include(action.g)`, wire `ActionRuleplgNow`, migrate `Parse/Grammar/plg.act` to the
new format, and confirm `plg plg.g` self-hosts. Until then, `attachActions` (§5) is the
working bootstrap.

⚠️ **Pending code change (not just a design note):** `action.g`'s `ActionEnd` rule still
matches `'end'`. It must be updated to **`'enD'`** to match the terminator the migration and
`attachActions` now use (§4). Until that edit lands, the grammar-driven path and the
bootstrap parser disagree on the terminator. This is a concrete one-line fix to make as part
of enabling `action.g`.

## 7. The PLGitem surface (post Phase-Integrate)

Current `PLGitem` (in `Parse/PLGitem.twk`; external decl in `PLGrevision`):

- Fields: `start`/`length`/`label`, `children` (BaseHash), `terminator`, `next`, `value`
  (void\*), `conditionResult`, `deferred`, `deferRule`, `savedChar`, and **`amount`** (see
  below).
- Methods: `compare`, `string` / `unString` / `toString`, **`getLabel(name)`**,
  `runDeferred(state)`, **`getAmount()`**.
- **Removed** (these break old action bodies — the Phase-Integrate migration updated the
  `.twk` files but originally missed the action code): the old `get`/`[]` accessors, the
  `test` field (the PLGtester back-pointer), the `testParser` field. Mappings:
  - `iTEM.get("L")` → `iTEM.children["L"]`
  - `iTEM["L"]` → still works: **`overload [] getLabel`** is declared in PLGrevision's
    `external PLGitem`, so the subscript form resolves to `getLabel` (→ `children[…]`).
    Only PLGitem `[]` is overloaded — KeyTable/BaseHash/Stak subscripts are unaffected. Haps edit: KeyTable, BaseHash, and a couple of other support classes also overload [] and []=; not an issue if they are not used. The overloads for those classes are defined in the tok external file frame that lives in the Include directory (I am not sure where to find it in the repos)
  - `iTEM.testParser.X` → bare `X` (action methods are on the parser).
  - `iTEM.test.X` → gone; review case-by-case (one site was simply removable).

**The `amount` decision (confirmed by Tony + Clay):** `amount` is a **real settable field**
on PLGitem (e.g. an array dimension the parser computes and stores — there are explicit
`array.amount = i;` write sites, so it is *not* vestigial). **`getAmount()`** is the read
accessor for the *text-derived* case (it returns `atoi(toString())`, replacing the old
auto-`atoi`-on-numeric-match behaviour). **Both coexist**: use the field when reading a
stored value, `getAmount()` when the numeric value should be recomputed from the matched
text.

## 8. Build workflow + gotchas

```bash
tok PLGparse.twk plgDirectives          # regenerate .C/.h for a changed runtime class
tok PLGitem.twk plgDirectives
xcodebuild -project Parse/plg.xcodeproj -target plg -configuration Debug
plg Tawk.g                              # (in Tokf/Tests) regenerate Tawk.twk
tok Tawk.twk                            # compile the generated transpiler
```

- **Debug config only.** plg's Release config is broken (the `support` dependency can't
  find `PLGparse.h`).
- **Always pass `plgDirectives`** to `tok` for plg's `.twk` files, or directive-injected code is silently stripped.
- `~/bin/plg` is a symlink → `Parse/build/Debug/plg`. Rebuild after any change to a runtime `.twk` that plg uses.
- A change to a `PLGrevision` external block (e.g. an overload) is consumed by `tok` when it compiles a *consumer* file (like `Tawk.twk` via `include includes`) — no plg rebuild needed for that effect, though plg's own files include PLGrevision too.

## 9. The grammar-drift problem (why this matters)

During the May refactor, plg was revised by hand-editing the generated `PLG.twk` and the runtime classes directly; `plg.g`/`plg.rtn`/`plg.act` were **not** kept in step, so they'd regenerate an *old* plg. Treat the grammar sources as suspect until a `plg plg.g` round-trip is proven. The action.g/named-options pass (§11) is where this gets reconciled.

## 11. PENDING: the named-options design pass

The one piece deliberately parked. `writeCaptures` currently injects labels from **all** alternatives of a rule. For a multi-alternative rule this over-injects: e.g. `FieldBody` has 3 alternatives (`prefix`,`part` / `name` / `name`), so it injects a `name` capture that collides with the body's own `PLGitem name = part["name"];`. The template injected only the relevant option's labels.

The correct fix is **per-option capture injection** — option N's action gets alternative N's labels — which *is* the named-options design (how options map to rules/alternatives, and what the option-tag syntax should be). It can't be solved cleanly without that design.
The named-options pass should: design the option syntax in `plg.g` + `action.g` together, make `writeCaptures` per-option, enable `action.g` (§6), migrate `plg.act`, and prove `plg plg.g` self-hosts. The ~10 remaining `tok Tawk.twk` FAILs are all this collision.
